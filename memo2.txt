直接回答你的问题：不可以。
你代码中使用的 uiautomation 库（全名 yinkaisheng/Python-UIAutomation-for-Windows）是Windows 专用的。它底层调用的是 Windows 的 Accessibility API (UI Automation)，在 macOS 上完全无法运行。
如果你想在 macOS 上实现类似功能，有以下几种方案：
方案一：使用 macOS 的 UI 自动化库（最接近你现在的写法）
在 macOS 上，类似的底层机制叫做 Accessibility API。Python 有对应的库 atomacos 可以封装这个功能。
步骤：
安装库：pip install atomacos
重要前提：你需要给终端（Terminal/iTerm）和 VS Code 开启 macOS 的“辅助功能”权限（系统设置 -> 隐私与安全性 -> 辅助功能）。
Mac 版代码示例（逻辑迁移）：

import atomacos
import time

def find_specific_text_mac(element, target_keyword, depth):
    if depth > 15:
        return None

    try:
        # 获取子元素列表
        children = element.AXChildren
        if not children:
            return None

        for child in children:
            try:
                # 获取元素的值或标题
                # Mac 上文本内容通常在 AXValue 或 AXTitle 属性中
                text_content = getattr(child, 'AXValue', '') or getattr(child, 'AXTitle', '')

                # 判断逻辑
                if text_content and target_keyword in str(text_content):
                    return child

                # 递归
                result = find_specific_text_mac(child, target_keyword, depth + 1)
                if result:
                    return result
            except:
                continue # 忽略无法访问的元素
    except:
        return None
    return None

方案二：使用 Playwright 连接调试端口（强烈推荐 🔥）
VS Code 本质上是一个浏览器（Electron 框架）。比起用操作系统的 UI 树（Windows UI Automation 或 Mac Accessibility）去“盲人摸象”，直接通过浏览器调试协议（CDP）去获取 DOM 结构是跨平台且最稳定的方法。
这种方法 Windows 和 Mac 通用代码，且速度极快。
步骤：
启动 VS Code 时开启调试端口：
在终端中通过命令启动 VS Code：

Bash
code --remote-debugging-port=9222
安装 Playwright：pip install playwright

from playwright.sync_api import sync_playwright

def main():
    with sync_playwright() as p:
        try:
            # 连接到已经运行的 VS Code 实例
            browser = p.chromium.connect_over_cdp("http://localhost:9222")

            # VS Code 有很多后台页面（Context），我们需要找到插件显示的那个页面
            # 获取所有上下文
            context = browser.contexts[0]

            # 遍历所有页面/视图
            target_page = None
            for page in context.pages:
                # 这里打印一下标题帮助你确认是哪一个，通常 Codex 面板会有特定的 Title 或 URL
                print(f"检查页面: {page.title()}")

                # 尝试获取包含关键词的元素
                # 假设你要找的内容在某个 div 或 span 里
                try:
                    if page.get_by_text("Hey there! Ready when you are").count() > 0:
                        target_page = page
                        break
                except:
                    continue

            if target_page:
                print("✅ 找到 Codex 页面！")
                # 直接使用 CSS 选择器或文本定位，不再需要递归
                element = target_page.get_by_text("Hey there! Ready when you are").first
                print(f"内容: {element.inner_text()}")
            else:
                print("❌ 未在当前打开的视图中找到目标文本")

        except Exception as e:
            print(f"连接失败: {e}")
            print("请确保通过命令行启动 VS Code: code --remote-debugging-port=9222")

if __name__ == "__main__":
    main()

这是一个非常专业且敏锐的问题！
简短回答：
插件的 UI 并不完全物理隔离，但存在 DOM 隔离。
大部分像 Codex、Copilot Chat、ChatGPT 这样有复杂界面的插件，使用的是 Webview 技术。它们本质上是在 VS Code 的主界面里嵌了一个 <iframe>。
这意味着：
直接找找不到：你在主 Document 下用 querySelector 查不到插件里的内容。
但是可以拿到：只要你穿透这个 iframe（切换上下文），里面的 DOM 结构是一览无余的。
详细分析：插件 UI 的两种存在形式
为了拿到内容，你需要先判断这个插件属于哪一种：
1. 原生 UI 组件 (Native UI / Tree View)
如果插件只是在侧边栏显示一个简单的列表（比如文件列表、大纲、简单的项目管理器），它们通常是 VS Code 原生渲染的。
隔离情况：无隔离。
获取方式：它们直接存在于主 DOM 树中。你可以直接通过文本或 CSS 选择器定位到。
2. Webview 面板 (复杂插件，大概率 Codex 是这种)
如果插件界面看起来很花哨（有聊天气泡、自定义按钮、Markdown 渲染），它一定是一个 Webview。
隔离情况：Iframe 隔离。VS Code 会生成一个 <iframe> 标签把插件内容包起来。
获取方式：你需要先定位到这个 iframe，然后“进入”内部文档 (ContentDocument) 才能获取值。
如何验证 Codex 是不是 Webview？
在 VS Code 中打开 Codex 面板。
Help -> Toggle Developer Tools。
点击调试工具左上角的“小箭头”（元素选择器）。
点击 Codex 面板里的文字。
观察 Elements 面板：
如果你发现你点击的元素被包裹在一个 <div class="webview ..."> 或者直接是一个 <iframe> 标签下面。
或者你在 DOM 树里看到了 #document (iframe 里的文档)。
结论：它是 Webview，需要穿透 iframe。
如果是 Webview，如何用 Python (Playwright) 获取？
接着上一个 Playwright 的方案，如果目标在 Webview (Iframe) 里，代码需要微调：

from playwright.sync_api import sync_playwright

def main():
    with sync_playwright() as p:
        try:
            # 连接 VS Code (记得先用命令行启动: code --remote-debugging-port=9222)
            browser = p.chromium.connect_over_cdp("http://localhost:9222")
            context = browser.contexts[0]

            # 1. 找到 VS Code 主页面
            page = context.pages[0]

            print("正在寻找 Codex 的 Webview iframe...")

            # 2. 定位 iframe
            # VS Code 的 Webview iframe 通常没有固定的 ID，但可以通过 src 或 class 寻找
            # 这里使用 frame_locator 自动穿透 iframe

            # 策略：在页面上所有的 iframe 中查找包含特定文字的那个
            # 注意：VS Code 的 Webview 可能会嵌套 iframe，所以我们要找到承载内容的那个

            codex_frame = None

            # 遍历所有的 frame (包括主页面和所有子 iframe)
            for frame in page.frames:
                try:
                    # 试探性查找：看看这个 frame 里有没有 Codex 的特征文字
                    # 比如聊天记录里的 "Hey there" 或者输入框的 placeholder
                    if frame.get_by_text("Hey there").count() > 0:
                        codex_frame = frame
                        print(f"✅ 锁定目标 Iframe: {frame.url}")
                        break
                except:
                    pass

            if codex_frame:
                # 3. 在 iframe 内部获取内容
                # 就像操作普通页面一样
                text_element = codex_frame.get_by_text("Hey there").first
                print("="*20)
                print(f"抓取到的内容: {text_element.inner_text()}")
                print("="*20)

                # 甚至可以获取输入框输入内容
                # input_box = codex_frame.locator('textarea')
                # input_box.fill("你好")
            else:
                print("❌ 未找到包含目标文本的 Webview (Iframe)。")

        except Exception as e:
            print(f"出错: {e}")

if __name__ == "__main__":
    main()
总结
VS Code 插件不是黑盒：它们本质上就是 HTML/CSS/JS。
障碍是 Iframe：所谓的“隔离”只是为了防止样式冲突和安全问题加的一层 iframe。
DOM 完全可拿：
Mac 系统级 (Accessibility API)：因为 iframe 的存在，Mac 的 Accessibility Inspector 往往很难深入到 iframe 内部读取内容（这解释了你为什么用 uiautomation 或 atomacos 可能读不到）。
DOM 级 (Playwright/CDP)：这是唯一靠谱的方案，因为浏览器调试协议天然支持跨 iframe 调试。

这是一个非常实用的需求。鉴于你希望代码具有跨平台（Windows/Mac）、自动化（滚动加载）、易集成（封装成类）的特性，我们继续采用 Playwright + CDP 的方案。
这是目前唯一能稳定穿透 VS Code 的 Webview (iframe) 并控制滚动条的方案。
核心功能特点
自动连接：连接到开启了调试端口的 VS Code。
智能定位：自动遍历所有 iframe，寻找包含 Codex 特征的 Webview。
逆向滚动：模拟鼠标滚轮向上滚动，触发 VS Code 的“懒加载”机制以获取历史消息。
结构化导出：将对话提取为 List/JSON 格式。

Python
import time
import json
from typing import List, Dict, Optional
from playwright.sync_api import sync_playwright, Page, Frame, Locator

class VSCodeCodexScraper:
    def __init__(self, port: int = 9222):
        self.port = port
        self.browser = None
        self.playwright = None
        self.target_frame: Optional[Frame] = None

    def connect(self):
        """连接到 VS Code 实例"""
        try:
            self.playwright = sync_playwright().start()
            self.browser = self.playwright.chromium.connect_over_cdp(f"http://localhost:{self.port}")
            print(f"✅ 成功连接到 VS Code (Port {self.port})")
        except Exception as e:
            print(f"❌ 连接失败: {e}")
            print("请确保 VS Code 已通过命令行启动: --remote-debugging-port=9222")
            raise e

    def find_codex_webview(self, keyword="Codex") -> bool:
        """
        在所有上下文中寻找包含特定关键词的 Webview (iframe)。
        VS Code 的插件通常运行在 iframe 中。
        """
        # 遍历所有上下文（Context）和页面（Page）
        for context in self.browser.contexts:
            for page in context.pages:
                # 遍历页面内的所有 iframe
                for frame in page.frames:
                    try:
                        # 策略：检测 frame 内部是否包含 Codex 相关的特定元素或文本
                        # 这里我们用一种通用的方式：检查 frame 标题或内部文本
                        # 注意：Codex 插件的具体特征可能需要根据实际 DOM 微调

                        # 1. 尝试通过 URL 或 Name 过滤 (VS Code webview url 通常很长)
                        if "webview" not in frame.url:
                            continue

                        # 2. 尝试获取内容判断
                        # 假设 Codex 界面里一定有某些特征词，比如 "Codex" 或者输入框
                        # 你可以根据实际情况修改这个选择器，比如 '.chat-input'
                        content = frame.content()
                        if keyword in content or "chat" in content.lower():
                            self.target_frame = frame
                            print(f"✅ 锁定目标 Webview: {frame.url[:50]}...")
                            return True
                    except Exception as e:
                        continue

        print("❌ 未找到目标 Codex 面板，请确认面板已在 VS Code 中打开。")
        return False

    def scroll_to_load_history(self, max_scrolls=10):
        """
        模拟滚动以触发懒加载 (Lazy Loading)。
        通常聊天记录需要向上滚动才能加载更多。
        """
        if not self.target_frame:
            raise Exception("未选定目标 Frame")

        print("🔄 开始滚动加载历史记录...")

        # 定位滚动容器。
        # 注意：不同的插件 HTML 结构不同。
        # 大多数 VS Code 聊天插件的主容器是 body 或某个 class 为 scrollable 的 div
        # 这里我们尝试直接滚动 body
        scroll_selector = "body"

        previous_height = 0

        for i in range(max_scrolls):
            # 获取当前高度
            current_height = self.target_frame.evaluate("document.body.scrollHeight")

            # 策略：先滚到顶部触发加载，再滚到底部
            # 1. 滚到顶部 (加载旧消息)
            self.target_frame.evaluate("window.scrollTo(0, 0)")
            time.sleep(1) # 等待 DOM 渲染

            # 2. 滚到底部 (为了截图或保持视觉一致，可选)
            # self.target_frame.evaluate("window.scrollTo(0, document.body.scrollHeight)")

            new_height = self.target_frame.evaluate("document.body.scrollHeight")

            if new_height == previous_height:
                # 高度不再变化，说明可能加载完了
                print(f"🛑 滚动停止，高度未变化 (迭代: {i})")
                break

            previous_height = new_height
            print(f"📜 第 {i+1} 次滚动加载完成...")

    def extract_messages(self) -> List[Dict]:
        """
        提取对话内容。
        需要根据 Codex 具体的 HTML 结构调整 CSS 选择器。
        """
        if not self.target_frame:
            raise Exception("未选定目标 Frame")

        messages = []

        # === 核心选择器配置 (需要根据 Inspect 结果微调) ===
        # 假设：聊天气泡通常是 div，可能包含 class 'message', 'chat-item' 等
        # 如果你不知道具体 class，可以提取所有文本段落

        # 方案 A: 通用抓取 (抓取所有非空文本块)
        # elements = self.target_frame.locator("div, p, span").all()

        # 方案 B: 针对性抓取 (假设结构如下，需要你用 Developer Tools 确认一下)
        # 这里的 selector 是基于常见的 Chat UI 猜测的
        message_elements = self.target_frame.locator(".conversation-message, .chat-message, .message-content, [role='listitem']").all()

        if not message_elements:
            print("⚠️ 未通过特定选择器找到消息，尝试提取全部可见文本...")
            full_text = self.target_frame.locator("body").inner_text()
            return [{"type": "raw", "content": full_text}]

        print(f"🔍 找到 {len(message_elements)} 条消息元素，正在解析...")

        for idx, el in enumerate(message_elements):
            try:
                text = el.inner_text()
                if not text.strip():
                    continue

                # 尝试判断是谁发的 (通过 class 或位置)
                # 比如 class 包含 'user' 或 'me'
                class_attr = el.get_attribute("class") or ""
                sender = "User" if "user" in class_attr.lower() or "right" in class_attr.lower() else "AI"

                messages.append({
                    "id": idx,
                    "sender": sender,
                    "content": text,
                    "raw_class": class_attr
                })
            except:
                continue

        return messages

    def close(self):
        if self.browser:
            self.browser.close()
        if self.playwright:
            self.playwright.stop()

# === 使用示例 ===
def main():
    scraper = VSCodeCodexScraper()

    try:
        # 1. 连接
        scraper.connect()

        # 2. 定位 (确保你在 VS Code 已经打开了 Codex 面板)
        # 你可以传入你在面板上看到的独特关键词，比如 "Ask Codex" 或 "Type here"
        if scraper.find_codex_webview(keyword="Hey"):

            # 3. 滚动加载 (可选)
            scraper.scroll_to_load_history(max_scrolls=3)

            # 4. 导出
            data = scraper.extract_messages()

            # 5. 保存或打印
            print("\n" + "="*40)
            print(json.dumps(data, indent=2, ensure_ascii=False))
            print("="*40)

            # 保存到文件
            with open("codex_chat_history.json", "w", encoding="utf-8") as f:
                json.dump(data, f, indent=2, ensure_ascii=False)
                print(f"📁 已保存 {len(data)} 条消息到 codex_chat_history.json")

    except Exception as e:
        print(f"程序执行出错: {e}")

下面从原理和实战角度详细对比一下：
1. 为什么 PyChromeDevTools 看起来“获取不到” iframe？
PyChromeDevTools 只是对 CDP (Chrome DevTools Protocol) 的一层裸封装。
在 CDP 协议的原生设计中，DOM 树默认是不穿透 iframe 的。
当你发送 DOM.getDocument 时，它只能拿到主页面的 HTML。
遇到 <iframe> 标签，它只当成一个普通标签，不会自动进去。
如果你非要用 PyChromeDevTools 获取 iframe 内容，你需要手动完成以下步骤（非常底层）：
调用 Page.getFrameTree 获取所有 frame 的 ID。
使用 Target.attachToTarget 手动连接到那个 iframe 的 Session（CDP 的多目标机制）。
在新的 Session 里发送 DOM.getDocument。
自己处理所有的异步通信和 ID 映射。
PyChromeDevTools 代码大概长这样（伪代码，反人类）：

2. 为什么 Playwright 是降维打击？

自动穿透：不需要关心 frameId，直接像找 div 一样找 frame。

def diagnose_vscode():

def inspect_inner_structure():
    # 你的目标 UUID (外壳)
    target_uuid = "51d42127-1bc9-4415-ad11-e92c53e6f7a4"

    with sync_playwright() as p:
        browser = p.chromium.connect_over_cdp("http://localhost:9222")

        print(f"🎯 进入外壳 Frame: {target_uuid}")
        # 1. 定位到第一层 (外壳)
        outer_frame = page.frame_locator(f'iframe[name="{target_uuid}"]')

        # === 诊断步骤 A: 看看外壳里有什么 HTML ===
        # 我们打印 HTML 源码，而不是纯文本，这样能看到里面的标签
        outer_html = outer_frame.locator("body").inner_html()

        print("-" * 50)
        print(outer_html[:500]) # 只打印前500字符避免刷屏
        print("-" * 50)

        # === 诊断步骤 B: 尝试自动穿透到内部子 Frame ===
        # 检查外壳里是否有子 iframe
        child_frames = outer_frame.locator("iframe")
        count = child_frames.count()

        if count > 0:
            print(f"\n⚡ 发现内部还有 {count} 个子 Iframe！正在尝试穿透...")

            # 通常第一个就是真正的内容
            inner_frame = outer_frame.frame_locator("iframe").first

            # 获取内部的文本
            inner_text = inner_frame.locator("body").inner_text()

            print("\n✅ 成功获取内部 Webview 数据：")
            print("=" * 50)
            print(inner_text)
            print("=" * 50)
        else:
            print("\n❌ 内部没有子 iframe，可能是 Shadow DOM 或内容未加载。")

if __name__ == "__main__":
    inspect_inner_structure()

这是一个非常典型的 "iframe 加载时序" 或 "上下文未就绪" 问题。
错误分析：
Timeout 30000ms exceeded: 说明 Playwright 找到了内部的 <iframe> 标签，但是无法连接到这个 iframe 内部的 文档 (Document)。
原因：VS Code 的 Webview 内部通常包含一个 id="active-frame" 的子 iframe。这个子 iframe 是由外壳（你刚才获取到 HTML 的那一层）里的 JS 动态创建和加载的。Playwright 在它还没完全加载好（或者还是 about:blank 状态）的时候就去读 body，所以超时了。
解决方案：
不要使用 .frame_locator().frame_locator() 这种硬连写（容易死锁）。
我们改用 "扁平化遍历" 策略。在 CDP 协议中，所有的 Frame 其实都是平铺的列表，我们只需要找到 "父节点是你那个 UUID" 的子 Frame 即可。
请运行这个修复版脚本，它包含自动等待逻辑：
code
Python
import time
from playwright.sync_api import sync_playwright

def solve_nested_iframe():
    # 你的目标 UUID (外壳)
    target_uuid = "51d42127-1bc9-4415-ad11-e92c53e6f7a4"

    with sync_playwright() as p:
        print("🔗 连接中...")
        browser = p.chromium.connect_over_cdp("http://localhost:9222")
        page = browser.contexts[0].pages[0]

        # 1. 先找到外壳 Frame 对象
        # 注意：这里我们不是要在页面里“找”它，而是直接在内存里的 frames 列表中“选”它
        outer_frame = None
        for f in page.frames:
            if f.name == target_uuid:
                outer_frame = f
                break

        if not outer_frame:
            print(f"❌ 找不到名为 {target_uuid} 的外壳 Frame。ID 可能变了？")
            return

        print(f"✅ 找到外壳 Frame: {outer_frame.url[:50]}...")

        # 2. 寻找外壳的“亲生子” Frame (这就是真正的内容)
        print("⚡ 正在寻找内部子 Frame (Active Frame)...")

        inner_frame = None
        # 最多尝试 10 次，每次等待 1 秒 (处理动态加载)
        for i in range(10):
            # 重新扫描所有 frame，寻找 parent 是 outer_frame 的那个
            child_frames = [f for f in page.frames if f.parent_frame == outer_frame]

            if child_frames:
                inner_frame = child_frames[0]
                print(f"🎉 找到了子 Frame! URL: {inner_frame.url[:50]}...")
                break
            else:
                print(f"   ⏳ 等待子 Frame 加载... ({i+1}/10)")
                time.sleep(1)

        if inner_frame:
            # 3. 等待内容就绪
            print("⏳ 等待 DOM 渲染...")
            try:
                # 显式等待 body 元素出现
                inner_frame.wait_for_selector("body", timeout=5000)

                # 获取内容
                content = inner_frame.inner_text("body")

                print("\n" + "="*50)
                print("🏆 终于拿到了！完整内容如下：")
                print("="*50)
                print(content)
                print("="*50)

                # 顺便打印一下 HTML 结构，方便你下次写选择器
                # print("Debug HTML片段:", inner_frame.content()[:500])

            except Exception as e:
                print(f"❌ 子 Frame 找到了，但读取超时: {e}")
                print("可能是 Webview 跨域限制，或者内容也是空的。")
        else:
            print("❌ 超时：外壳里一直没有加载出子 Frame。")

if __name__ == "__main__":
    solve_nested_iframe()
为什么这个脚本能行？
逻辑改变：之前的脚本是 outer.frame_locator("iframe")，这要求 DOM 树必须严格层级对应。现在的脚本是 page.frames，直接在内存里找所有已加载的 frame，然后通过 parent_frame 属性反向匹配。
加入重试：VS Code 的内部 iframe 是异步加载的（JS 执行后才插入 DOM）。之前的脚本跑太快了，现在加了 for i in range(10) 循环等待。
避免死锁：直接操作 Frame 对象比操作 FrameLocator 更底层、更稳定。

这是一个非常典型的**“找错房间”**的问题。
你说 ID 没变，我完全相信你。之前的脚本报错是因为它只在 browser.contexts[0].pages[0]（第一个标签页）里找。
但 VS Code 是多进程应用，当你打开开发者工具或其他插件时，这个 Frame 可能跑到了 pages[1] 或者其他 Context 里，导致脚本在 pages[0] 里遍历不到它。
请运行下面这个**“全剧搜捕”**脚本。它不假设 Frame 在哪个页面，而是遍历 VS Code 的所有角落去找这个 UUID。

import time
from playwright.sync_api import sync_playwright

def global_search_and_extract():
    # 你的目标 UUID
    target_uuid = "51d42127-1bc9-4415-ad11-e92c53e6f7a4"

    with sync_playwright() as p:
        print("🔗 连接 VS Code (9222)...")
        browser = p.chromium.connect_over_cdp("http://localhost:9222")

        target_frame = None
        found_in_page = None

        # === 步骤 1: 全局扫描所有 Context 和 Page ===
        print(f"🔎 开始在所有页面中搜捕 Frame: {target_uuid}")

        for ctx_idx, context in enumerate(browser.contexts):
            for page_idx, page in enumerate(context.pages):
                # 遍历这个页面里所有的 frame
                for frame in page.frames:
                    # 检查 frame.name 是否匹配
                    # 或者检查 frame 对应的 HTML 元素是否有这个 name 属性
                    if frame.name == target_uuid:
                        target_frame = frame
                        found_in_page = page
                        print(f"✅ 找到了！")
                        print(f"   📍 位置: Context {ctx_idx} -> Page {page_idx} ('{page.title()}')")
                        break
                if target_frame: break
            if target_frame: break

        if not target_frame:
            print("❌ 彻底搜索了所有页面，仍然未找到该 UUID。")
            print("可能是：")
            print("1. 你重启了 VS Code（UUID 肯定会变）。")
            print("2. 你关闭了 Codex 面板。")
            return

        # === 步骤 2: 穿透到内部 ===
        print("\n⚡ 正在分析内部结构...")

        # 尝试获取子 Frame
        # child_frames 属性包含了所有直接子 frame
        children = target_frame.child_frames
        print(f"   该 Frame 包含 {len(children)} 个子 Frame")

        real_content_frame = None

        if len(children) > 0:
            real_content_frame = children[0]
            print(f"   👉 锁定第一个子 Frame: {real_content_frame.url[:50]}...")
        else:
            print("   ⚠️ 没有检测到子 Frame (可能是加载延迟)，尝试等待 3 秒...")
            time.sleep(3)
            # 重新获取
            children = target_frame.child_frames
            if len(children) > 0:
                real_content_frame = children[0]
                print(f"   👉 重试后锁定子 Frame: {real_content_frame.url[:50]}...")
            else:
                print("   ❌ 依然没有子 Frame，尝试直接读取外壳内容...")
                real_content_frame = target_frame

        # === 步骤 3: 暴力读取 ===
        print("\n📖 正在读取内容...")
        try:
            # 确保 body 加载出来
            real_content_frame.wait_for_selector("body", timeout=5000)

            # 获取文本
            content = real_content_frame.inner_text("body")

            print("\n" + "="*50)
            print("🎉 最终内容:")
            print("="*50)
            print(content)
            print("="*50)

        except Exception as e:
            print(f"❌ 读取失败: {e}")
            print("建议：如果内容为空，可能是 VS Code 的 Webview 跨域限制，需要用更底层的 CDP 方法，但先看这次结果。")

if __name__ == "__main__":
    global_search_and_extract()

这种情况非常诡异，通常意味着 Playwright 连接到的“世界”和你在屏幕上看到的“世界”不是同一个。
VS Code 是一个多进程架构（Electron），有时候 connect_over_cdp 默认连上的只是主进程，而 Webview 跑在另一个独立的渲染进程里，导致 Playwright 的 page.frames 列表里根本没有那个 Frame。
请做以下两步操作，我们要进行底层的资产清点。
第一步：物理确认（非常重要）
请在你的**普通浏览器（Chrome/Edge）**中输入以下地址：
http://localhost:9222/json
你会看到一堆 JSON 数据。请搜索你那个 ID 51d42127...。
如果搜不到：说明这个 ID 在调试端口里根本不存在（VS Code 绝对悄悄换了 ID，或者该 Webview 还没注册到调试协议里）。
如果搜到了：请看它的 type 是什么（是 page 还是 webview 还是 iframe）。

from playwright.sync_api import sync_playwright

def inventory_check():
    # 你的目标 ID
    target_id = "51d42127-1bc9-4415-ad11-e92c53e6f7a4"

    with sync_playwright() as p:
        print("🔗 连接 VS Code (9222)...")
        browser = p.chromium.connect_over_cdp("http://localhost:9222")

        print(f"\n📋 开始资产盘点 (寻找目标: {target_id})")
        print("=" * 60)

        found = False
        frame_count = 0

        # 遍历所有 Context
        for i, ctx in enumerate(browser.contexts):
            # 遍历所有 Page
            for j, page in enumerate(ctx.pages):
                print(f"\n📄 [Context {i} | Page {j}] 标题: {page.title()}")

                # 遍历所有 Frame
                for frame in page.frames:
                    frame_count += 1
                    # 获取 frame 的 name 属性（这是 Playwright 认为的名字）
                    p_name = frame.name

                    # 尝试获取 HTML 标签上的 name 属性 (这是 Inspector 看到的名字)
                    # 有时候 frame.name 是空的，但 HTML 标签里有 name
                    try:
                        html_name = frame.evaluate("window.name")
                    except:
                        html_name = "无法读取"

                    url_preview = frame.url[-40:] if len(frame.url) > 40 else frame.url

                    # 打印匹配情况
                    is_match = (target_id == p_name) or (target_id == html_name)
                    marker = "✅✅✅ 找到了！！！" if is_match else ""

                    if is_match: found = True

                    print(f"   Frame: Name='{p_name}' | HTML_Name='{html_name}' | URL=...{url_preview} {marker}")

        print("=" * 60)
        print(f"统计: 共扫描了 {frame_count} 个 Frame。")

        if found:
            print("结论: Playwright 能看到它！之前的脚本逻辑有问题。")
        else:
            print("结论: Playwright 根本看不到这个 Frame。")
            print("可能原因：")
            print("1. 它是 Webview 进程，需要独立连接（见下一步）。")
            print("2. ID 真的变了（Inspector 里的信息可能是旧的缓存）。")

if __name__ == "__main__":
    inventory_check()

如果上面的脚本找不到，说明 Codex 的 Webview 是作为一个**独立的目标（Target）**存在的，而不是主页面的子 Frame。

import requests
from playwright.sync_api import sync_playwright

def connect_direct_target():
    print("🕵️‍♂️ 尝试直接扫描 CDP Targets...")

    # 1. 直接问 VS Code 的调试端口：你有几个目标？
    try:
        response = requests.get("http://localhost:9222/json")
        targets = response.json()
    except Exception as e:
        print(f"无法连接调试端口: {e}")
        return

    target_ws = None

    # 2. 在 targets 里找 Webview
    for t in targets:
        title = t.get('title', '')
        url = t.get('url', '')

        # 打印出来看看
        # print(f"发现 Target: {title} - {url}")

        # 寻找疑似 Codex 的 Target
        if "webview" in url or "Codex" in title or "extension" in url:
            print(f"🎯 发现疑似目标 Target: {title}")
            print(f"   URL: {url}")
            print(f"   WebSocket: {t.get('webSocketDebuggerUrl')}")

            # 如果这看起来像你要的 webview，我们记录它的 ws 地址
            # VS Code 的 Webview title 通常叫 "Webview" 或者插件名
            if "webview" in url:
                target_ws = t.get('webSocketDebuggerUrl')
                break

    if not target_ws:
        print("❌ 未找到独立的 Webview Target。")
        return

    # 3. 直接连接这个 Target
    print(f"\n🚀 正在通过 WS 直连 Target: {target_ws} ...")

    with sync_playwright() as p:
        # 注意：这里不是 connect_over_cdp (http)，而是 connect (ws)
        browser = p.chromium.connect(ws_endpoint=target_ws)
        page = browser.contexts[0].pages[0]

        print("✅ 连接成功！正在读取内容...")
        try:
            content = page.content() # 获取完整 HTML
            print(f"页面长度: {len(content)}")

            # 尝试找文字
            text = page.inner_text("body")
            print("\n内容预览:")
            print(text[:200])

        except Exception as e:
            print(f"读取失败: {e}")

if __name__ == "__main__":
    connect_direct_target()

JS 是运行在浏览器内部的，它能直接访问 window.frames 和 document，不需要经过 Playwright 的协议转换。

import time
from playwright.sync_api import sync_playwright

def get_codex_js_bypass():
    target_id = "2f171150-4bec-4222-ba0e-f29536ae8f19"

    with sync_playwright() as p:
        print("🔗 连接 VS Code...")
        browser = p.chromium.connect_over_cdp("http://localhost:9222")
        page = browser.contexts[0].pages[0]

        # === 1. 定位外壳 ===
        print(f"🎯 正在定位外壳 (ID: {target_id})...")
        shell_frame = None

        for _ in range(5):
            for frame in page.frames:
                try:
                    if frame.evaluate("window.name") == target_id:
                        shell_frame = frame
                        break
                except: continue
            if shell_frame: break
            time.sleep(1)

        if not shell_frame:
            print("❌ 找不到外壳。")
            return

        print("✅ 找到外壳 Frame。")

        # === 2. 诊断阶段：看看外壳里到底有什么 ===
        print("\n🏥 [诊断] 使用 JS 检查外壳内部状态:")
        try:
            # 检查1: iframe 标签数量
            iframe_count = shell_frame.evaluate("document.querySelectorAll('iframe').length")
            print(f"   👉 内部 <iframe> 标签数量: {iframe_count}")

            # 检查2: window.frames 对象数量 (JS视角的子窗口)
            window_frames_count = shell_frame.evaluate("window.frames.length")
            print(f"   👉 window.frames 长度: {window_frames_count}")

            # 检查3: 打印 body HTML (看看结构)
            body_html = shell_frame.evaluate("document.body.innerHTML")
            print(f"   👉 Body HTML 长度: {len(body_html)}")
            if len(body_html) < 500:
                print(f"   👉 Body HTML 内容: {body_html}")

        except Exception as e:
            print(f"   ❌ JS 执行出错: {e}")

        # === 3. 核心：JS 暴力提取 ===
        print("\n⚡ 正在尝试 JS 跨域穿透提取...")

        found_content = False

        for i in range(10):
            try:
                # 这段 JS 脚本尝试做两件事：
                # 1. 找到第一个 iframe
                # 2. 读取它内部 document 的 body.innerText
                # 注意：这里处理了跨域访问 (contentDocument 可能为 null)
                js_script = """
                    () => {
                        const iframe = document.querySelector('iframe');
                        if (!iframe) return { status: 'no_iframe' };

                        try {
                            const doc = iframe.contentDocument || iframe.contentWindow.document;
                            if (!doc) return { status: 'cross_origin_blocked' };

                            // 获取文本
                            const text = doc.body.innerText;
                            // 获取 HTML 以便调试
                            const html = doc.body.innerHTML;

                            return {
                                status: 'success',
                                text: text,
                                html_preview: html.substring(0, 100),
                                length: text.length
                            };
                        } catch (e) {
                            return { status: 'error', error: e.toString() };
                        }
                    }
                """

                result = shell_frame.evaluate(js_script)
                status = result.get('status')

                print(f"   [尝试 {i+1}] JS 返回状态: {status}")

                if status == 'success':
                    text = result.get('text', '')
                    if len(text.strip()) > 0:
                        print("\n" + "="*40)
                        print("🏆 终于成功了 (JS Bypass) 🏆")
                        print("="*40)
                        print(text)
                        print("="*40)
                        found_content = True
                        break
                    else:
                        print("   ⚠️ 成功进入内部，但 body.innerText 是空的 (可能还在渲染)")
                elif status == 'cross_origin_blocked':
                    print("   ❌ 跨域拦截: VS Code 阻止了直接访问 contentDocument")
                    # 如果走到这里，我们必须换用 ElementHandle + contentFrame() 的变种
                    break
                elif status == 'error':
                    print(f"   ❌ JS 内部错误: {result.get('error')}")

            except Exception as e:
                print(f"   ⚠️ Python 执行异常: {e}")

            time.sleep(1)

        if not found_content:
            print("\n❌ 所有尝试均失败。")

if __name__ == "__main__":
    get_codex_js_bypass()

import atomacos
import time
import sys

# === 核心递归查找函数 ===
def find_element_by_text(element, keyword, depth=0, max_depth=20):
    if depth > max_depth:
        return None

    try:
        # 获取子元素列表 (AXChildren)
        children = element.AXChildren
        if not children:
            return None

        for child in children:
            try:
                # 1. 获取属性
                role = getattr(child, 'AXRole', '')
                value = getattr(child, 'AXValue', '')
                title = getattr(child, 'AXTitle', '')

                # 2. 打印当前扫描到的元素 (调试用，不需要可以注释掉)
                # print(f"{'  ' * depth} [{role}] Value: {value} | Title: {title}")

                # 3. 判断逻辑 (检查 Value 或 Title 是否包含关键词)
                # Codex 的内容通常在 AXTextArea 或 AXStaticText 里
                content = str(value) if value else str(title)

                if keyword in content:
                    return child

                # 4. 递归查找
                # 优化：只递归容器类型的元素，跳过普通按钮以节省时间
                if role in ['AXGroup', 'AXWebArea', 'AXSplitGroup', 'AXScrollArea']:
                    result = find_element_by_text(child, keyword, depth + 1, max_depth)
                    if result:
                        return result
            except:
                continue
    except:
        return None

    return None

def main():

    # 1. 获取 VS Code 应用对象

        vscode_app = atomacos.getAppRefByBundleId('com.microsoft.VSCode')

        print("❌ 无法获取 VS Code 应用，请确认它已运行。")

    # 2. 获取主窗口

    main_window = windows[0]
    print(f"✅ 锁定主窗口: {main_window.AXTitle}")

    # 3. 开始查找

这种方法的局限性 (针对 Electron 应用)
虽然这是你要的“原生方案”，但我必须诚实告诉你它的坑：
Electron 的黑盒问题：
VS Code 是 Electron 应用。Electron 渲染的网页（Webview）在 Accessibility Inspector 里通常显示为一个巨大的 AXWebArea 或 AXGroup。
好情况：VS Code 把 DOM 树完整映射成了 AX 树，你可以像上面的代码一样找到文字。
坏情况：Codex 插件的内容被视为“画布”或“私有内容”，系统只能看到有一个框，读不到里面的字。
速度慢：
Accessibility API 的递归遍历比 DOM 遍历（Playwright）慢得多，因为它是通过进程间通信（IPC）获取数据的。
结论：
你可以先用 Accessibility Inspector 手动点一下 Codex 的文字。
如果 Inspector 能看到文字：那么上面的 atomacos 代码就能工作。
如果 Inspector 只能选中整个大框却看不到字：那么这条路走不通，只能回过头去解决 Playwright/CDP 的问题（它是直接读内存，不受 UI 映射限制）。

在 Windows 上，使用 inspect.exe 配合 Python 的 uiautomation 库是获取标准 UI 内容的最常用方法。
针对 VS Code 这种 Electron 应用（网页套壳），要获取 Codex 插件上的所有内容（包括历史记录），你需要解决两个难点：
定位 Webview：插件内容包裹在一个深层嵌套的容器里。
滚动加载：屏幕外的内容 VS Code 是不渲染的（虚拟列表），必须边滚动边抓取。
以下是完整的解决方案和代码。
前置准备：确认元素是否“可见”
在写代码前，请打开 Windows SDK 里的 inspect.exe：
鼠标指向 Codex 插件的聊天文字。
看 Inspect 左侧的树状图。
关键检查：如果你只能看到一层 Chrome_RenderWidgetHostHWND，而看不到里面的 Text 或 Group，说明 VS Code 没有向 Windows 暴露无障碍树。
解决方法：无需重启，只需在 VS Code 窗口还在的时候，让 Inspect 悬停一会，或者运行一遍下面的代码，Electron 通常会自动检测到辅助功能请求并开启 UI 树。

import sys
import time
import uiautomation as auto

# 设置全局超时，防止找不到元素时卡太久
auto.SetGlobalSearchTimeout(2)

class CodexScraper:
    def __init__(self):
        self.vscode = None
        self.codex_panel = None
        # 使用集合去重，防止滚动时重复抓取
        self.captured_messages = []
        self.seen_hashes = set()

    def connect_vscode(self):
        print("🔗 正在连接 VS Code...")
        # 正则匹配窗口标题
        self.vscode = auto.WindowControl(RegexName='.*Visual Studio Code')

        if not self.vscode.Exists(3):
            print("❌ 未找到 VS Code 窗口！")
            return False

        self.vscode.SetFocus()
        return True

    def find_codex_panel(self):
        print("🔍 正在定位 Codex 面板...")
        # 策略：Codex 面板通常是一个 DocumentControl 或 GroupControl
        # 我们在整个窗口里找 Name 包含 Codex 的控件
        # 注意：Depth 设置大一点，因为 Electron 层级很深

        # 1. 尝试找 Document (最常见)
        panel = self.vscode.DocumentControl(Name="Codex", SearchDepth=15)

        # 2. 如果找不到，尝试找包含 Codex 字样的 Group
        if not panel.Exists(1):
             # 使用 lambda 自定义查找条件，找 Name 包含 Codex 的任意控件
            panel = self.vscode.Control(
                searchDepth=15,
                compare=lambda c, d: "Codex" in c.Name if c.Name else False
            )

        if panel.Exists(1):
            print(f"✅ 找到面板: {panel.Name}")
            # 获取该面板的中心坐标，方便后续鼠标滚动
            self.codex_panel = panel
            return True
        else:
            print("❌ 未找到 Codex 面板。请确保插件已打开并显示在侧边栏或编辑器区域。")
            return False

    def _extract_visible_text(self):
        """提取当前视野内的所有文本控件"""
        # 获取面板下的所有文本子节点
        # 深度设为 10 足够穿透气泡包裹层
        text_controls = self.codex_panel.GetChildren()
        # 如果 GetChildren 拿不到深层的，可以用 GetFirstChild/GetNextSibling 递归，
        # 但 auto.WalkControl 或 FindAll 更方便

        # 使用 FindAll 查找所有 TextControl
        all_texts = self.codex_panel.FindAll(
            treeScope=auto.TreeScope.Descendants,
            controlType=auto.ControlType.TextControl
        )

        count = 0
        for ctrl in all_texts:
            text = ctrl.Name
            if not text or text.strip() == "":
                continue

            # 简单的去重逻辑 (基于内容哈希)
            # 实际聊天中可能存在完全相同的内容，建议加上坐标判断，这里简化处理
            msg_hash = hash(text)
            if msg_hash not in self.seen_hashes:
                self.seen_hashes.add(msg_hash)
                self.captured_messages.append(text)
                count += 1
                # 打印预览
                print(f"   [抓取] {text[:30]}..." if len(text)>30 else f"   [抓取] {text}")

        return count

    def scrape_all_content(self):
        if not self.codex_panel:
            return

        # 1. 鼠标移动到面板中心，确保滚动事件生效
        rect = self.codex_panel.BoundingRectangle
        center_x = (rect.left + rect.right) // 2
        center_y = (rect.top + rect.bottom) // 2
        auto.SetCursorPos(center_x, center_y)

        print("\n⬆️ 正在向上滚动以加载历史记录...")
        # 模拟滚轮向上 (WheelUp)，触发懒加载
        # 这里的次数取决于你的历史记录长度
        for i in range(10):
            auto.WheelUp(clicks=3)
            time.sleep(0.5) # 等待加载

        print("\n⬇️ 开始向下滚动并抓取内容...")

        no_new_content_count = 0

        while True:
            # 抓取当前屏幕内容
            new_items = self._extract_visible_text()

            if new_items == 0:
                no_new_content_count += 1
            else:
                no_new_content_count = 0 # 重置计数器

            # 如果连续 5 次滚动都没抓到新东西，说明到底了
            if no_new_content_count >= 5:
                print("🏁 似乎已到达底部或没有新内容。")
                break

            # 向下滚动
            auto.WheelDown(clicks=3)
            time.sleep(0.8) # 等待 UI 刷新

    def export(self):
        print("\n" + "="*40)
        print(f"📊 抓取完成！共 {len(self.captured_messages)} 条文本片段。")
        print("="*40)

        # 保存到文件
        with open("codex_full_export.txt", "w", encoding="utf-8") as f:
            for msg in self.captured_messages:
                f.write(msg + "\n" + "-"*20 + "\n")

        print("📁 内容已保存至 codex_full_export.txt")

def main():
    scraper = CodexScraper()
    if scraper.connect_vscode():
        if scraper.find_codex_panel():
            scraper.scrape_all_content()
            scraper.export()

if __name__ == '__main__':
    main()
关键点解析
FindAll(descendants):
VS Code 的 UI 结构非常深（Group 套 Group 套 Pane）。直接用 GetChildren() 只能拿到第一层。使用 FindAll 配合 TreeScope.Descendants 可以一次性把面板下的所有 TextControl 也就是聊天文字全部捞出来。
auto.WheelUp/WheelDown:
这是最核心的部分。VS Code 里的聊天列表是“虚拟列表”（Virtual List）。你用代码去遍历 DOM 树，只能读到当前屏幕上显示的那十几行字。必须模拟物理鼠标滚动，让 VS Code 卸载旧 DOM、加载新 DOM，我们才能抓取到完整对话。
去重逻辑 (seen_hashes):
因为我们是“滚一下、抓一下”，两次抓取之间肯定有重叠的内容。如果不去重，你的结果里会有大量重复的句子。代码里用了简单的哈希去重。
如果代码跑起来找不到元素 (常见坑)
如果你运行代码显示 ❌ 未找到 Codex 面板，或者找到了面板但是抓取数为 0，请检查：
Inspect 确认：用 inspect.exe 看一下，Codex 区域是不是一片空白（只有 Chrome_RenderWidgetHostHWND）。
强制刷新树：
VS Code 为了性能，有时默认不开启无障碍树。
技巧：在运行 Python 脚本前，先用 inspect.exe 选中一下 VS Code 的任意文字。这个动作会告诉 Electron “有读屏软件在运行”，从而强制它渲染完整的 UI 树。
总结：
Windows 上用 uiautomation 获取所有内容是可行的，但必须结合 鼠标模拟滚动 才能突破 Electron 的虚拟列表限制。

这是一个非常具体的 Tailwind CSS 类名。这表明我们可以通过 ClassName 精准定位到这个控件，而不需要依赖模糊的 Name。
Name 属性直接包含了内容：

注意看 Children 列表，里面的 "Cline has a question"、"你好..." 等子元素的文本，竟然全都被自动拼接到了父级 Group 的 Name 属性里！
这意味着：我们不需要遍历子元素，直接读取这个 Group 的 Name 属性，就能拿到整段对话！
✅ 针对 Cline 的定制抓取脚本
这个脚本利用你提供的 inspect.exe 信息，做了针对性优化：
定位策略：先找 Name="Cline" 的文档，再找 ClassName 包含 "scrollable" 的控件。
提取策略：直接读取 GroupControl 的 Name 属性（因为你发现它已经包含了所有文字）。
滚动策略：针对该控件进行滚动。

import sys
import time
import uiautomation as auto

# 设置超时
auto.SetGlobalSearchTimeout(3)

class ClineScraper:
    def __init__(self):
        self.vscode = None
        self.cline_doc = None  # Cline 的主文档区域
        self.captured_texts = []
        self.seen_hashes = set()

    def connect_vscode(self):
        print("🔗 正在连接 VS Code...")
        self.vscode = auto.WindowControl(RegexName='.*Visual Studio Code')
        if not self.vscode.Exists(3):
            print("❌ 未找到 VS Code 窗口")
            return False
        self.vscode.SetFocus()
        return True

    def find_cline_area(self):
        print("🔍 正在定位 Cline 面板...")

        # 1. 第一步：根据 Ancestors 信息，先找到 Name="Cline" 的 Document
        # Depth 设大一点，防止嵌套过深
        self.cline_doc = self.vscode.DocumentControl(Name="Cline", SearchDepth=20)

        if not self.cline_doc.Exists(1):
            # 备用：尝试模糊查找
            self.cline_doc = self.vscode.DocumentControl(
                searchDepth=20,
                compare=lambda c, d: "Cline" in c.Name if c.Name else False
            )

        if self.cline_doc.Exists(1):
            print(f"✅ 找到 Cline 文档区域: {self.cline_doc.Name}")

            # 2. 唤醒 UI 树 (点击中心)
            # Inspect 显示这通常是一个可滚动的区域
            print("👆 激活区域...")
            try:
                self.cline_doc.Click(simulateMove=True)
            except:
                rect = self.cline_doc.BoundingRectangle
                cx, cy = (rect.left + rect.right) // 2, (rect.top + rect.bottom) // 2
                auto.Click(cx, cy)

            return True
        else:
            print("❌ 未找到 Cline 文档。请确认侧边栏已打开 Cline。")
            return False

    def _extract_content(self):
        """
        根据 inspect.exe 发现的规律：
        目标内容的 ControlType 是 Group (组)
        ClassName 是 "scrollable grow overflow-y-scroll" (或者包含它)
        Name 属性直接包含了所有文本
        """
        found_count = 0

        # 遍历 Cline 文档下的所有控件
        # 我们寻找符合 inspect 特征的 GroupControl
        for ctrl, depth in auto.WalkControl(self.cline_doc, includeTop=False, maxDepth=15):

            # 过滤 1: 必须是 GroupControl (组)
            if ctrl.ControlTypeName != 'GroupControl':
                continue

            # 过滤 2: 只要 Name 有内容
            raw_text = ctrl.Name
            if not raw_text or len(raw_text.strip()) < 2:
                continue

            # 过滤 3 (可选): 根据 ClassName 精确匹配
            # 你的 inspect 显示 ClassName 是 "scrollable grow overflow-y-scroll"
            # 我们只要匹配 "scrollable" 或 "overflow" 即可
            # 注意：某些子气泡可能没有这个 class，所以这个条件可以视情况注释掉
            # if "scrollable" not in ctrl.ClassName and "overflow" not in ctrl.ClassName:
            #    continue

            # 去重处理
            text_hash = hash(raw_text)
            if text_hash not in self.seen_hashes:
                self.seen_hashes.add(text_hash)

                # 简单清洗：inspect 显示名字里包含一些 Unicode 图标 (如 )，可以不用管，或者替换掉
                clean_text = raw_text.strip()
                self.captured_texts.append(clean_text)
                found_count += 1

                print(f"   [抓取] {clean_text[:60].replace(chr(10), ' ')}...")

        return found_count

    def run(self):
        if not self.connect_vscode(): return
        if not self.find_cline_area(): return

        print("\n⬆️ 预滚动 (加载历史)...")
        # 对着 Cline 文档区域滚动
        # 注意：如果 document 不响应滚动，可能需要找到那个特定的 class="scrollable..." 的控件来滚动
        # 这里先尝试对着 Document 滚
        for _ in range(3):
            auto.WheelUp(wheelTimes=3)
            time.sleep(0.5)

        print("\n⬇️ 开始抓取...")
        no_new_data = 0

        while True:
            count = self._extract_content()

            if count == 0:
                no_new_data += 1
            else:
                no_new_data = 0

            # 连续 8 次没新内容才停止
            if no_new_data >= 8:
                print("🏁 抓取结束。")
                break

            # 向下滚动
            auto.WheelDown(wheelTimes=3)
            time.sleep(1.0) # 等待 Electron 渲染

    def save(self):
        print(f"\n📊 共抓取 {len(self.captured_texts)} 条记录。")
        if not self.captured_texts:
            return

        with open("cline_chat_history.txt", "w", encoding="utf-8") as f:
            for t in self.captured_texts:
                f.write(t + "\n" + "="*50 + "\n")
        print("📁 保存至 cline_chat_history.txt")

if __name__ == '__main__':
    s = ClineScraper()
    s.run()
    s.save()

Python
import time
import atomacos
from pynput.mouse import Controller, Button

class ClineMacScraper:
    def __init__(self):
        self.vscode = None
        self.window = None
        self.cline_group = None
        self.mouse = Controller()
        self.seen_hashes = set()
        self.captured_texts = []

    def connect_vscode(self):
        print("🔗 正在连接 VS Code (Mac)...")
        try:
            # Mac 使用 Bundle ID 连接
            self.vscode = atomacos.getAppRefByBundleId('com.microsoft.VSCode')
            # 获取主窗口
            windows = self.vscode.windows()
            if not windows:
                print("❌ 未找到 VS Code 窗口")
                return False
            self.window = windows[0]
            print(f"✅ 锁定窗口: {self.window.AXTitle}")
            return True
        except Exception as e:
            print(f"❌ 连接失败: {e}")
            print("⚠️ 请检查是否已授予终端/IDE“辅助功能”权限！")
            return False

    def _recursive_find(self, element, target_keyword, max_depth, current_depth=0):
        """模拟 Windows 的 Find/WalkControl"""
        if current_depth > max_depth:
            return None

        try:
            children = element.AXChildren
            if not children: return None

            for child in children:
                # 获取标题或描述
                # Mac 上的文本可能存在于 AXTitle, AXDescription 或 AXValue 中
                title = getattr(child, 'AXTitle', '')
                desc = getattr(child, 'AXDescription', '')

                # 检查是否匹配 "Cline"
                if (title and target_keyword in title) or (desc and target_keyword in desc):
                    return child

                # 递归查找 (只进入容器类型的元素，提高效率)
                role = getattr(child, 'AXRole', '')
                if role in ['AXGroup', 'AXSplitGroup', 'AXScrollArea', 'AXWebArea']:
                    found = self._recursive_find(child, target_keyword, max_depth, current_depth + 1)
                    if found: return found
        except:
            return None
        return None

    def find_cline_panel(self):
        print("🔍 正在递归搜索 Cline 面板 (可能需要几秒)...")
        # 在窗口里找包含 "Cline" 的元素
        self.cline_group = self._recursive_find(self.window, "Cline", max_depth=15)

        if self.cline_group:
            print(f"✅ 找到 Cline 区域: {getattr(self.cline_group, 'AXRole', 'Unknown')}")
            # 激活窗口
            self.vscode.activate()

            # 点击中心以激活焦点 (类似 Windows 的 Click)
            try:
                pos = self.cline_group.AXPosition
                size = self.cline_group.AXSize
                cx = pos.x + size.w / 2
                cy = pos.y + size.h / 2

                self.mouse.position = (cx, cy)
                self.mouse.click(Button.left, 1)
                print("👆 已点击激活区域")
            except:
                pass
            return True
        else:
            print("❌ 未找到 Cline 面板。")
            return False

    def _extract_content(self):
        """遍历当前区域的所有子元素提取文本"""
        found_count = 0
        try:
            # 类似 Windows 的 WalkControl
            # 这里我们只遍历 Cline 区域下的浅层子元素，或者你需要写递归遍历
            # 简单起见，这里假设 Cline 面板下的子 Group 就是消息气泡
            children = self.cline_group.AXChildren
            if not children: return 0

            for child in children:
                # 在 Windows 上，内容在 Name 属性里
                # 在 Mac 上，尝试获取所有可能的文本属性
                text_candidates = [
                    getattr(child, 'AXTitle', ''),
                    getattr(child, 'AXValue', ''),
                    getattr(child, 'AXDescription', '')
                ]

                # 拼接并清洗
                text = " ".join([str(t) for t in text_candidates if t]).strip()

                if not text or len(text) < 2: continue

                # 去重
                text_hash = hash(text)
                if text_hash not in self.seen_hashes:
                    self.seen_hashes.add(text_hash)
                    self.captured_texts.append(text)
                    found_count += 1
                    print(f"   [抓取] {text[:50].replace(chr(10), ' ')}...")

        except Exception as e:
            # 忽略瞬时 DOM 变化错误
            pass

        return found_count

    def run(self):
        if not self.connect_vscode(): return
        if not self.find_cline_panel(): return

        print("\n⬆️ 预滚动...")
        # Mac 的 scroll dy 正数是向上滚 (自然滚动设置下可能相反，视系统设置而定)
        # 这里假设非自然滚动，dy=2 向上滚
        for _ in range(5):
            self.mouse.scroll(0, 3)
            time.sleep(0.5)

        print("\n⬇️ 开始抓取...")
        no_new_data = 0

        while True:
            count = self._extract_content()

            if count == 0:
                no_new_data += 1
            else:
                no_new_data = 0

            if no_new_data >= 10:
                print("🏁 抓取结束。")
                break

            # 向下滚动 (dy 负数)
            self.mouse.scroll(0, -3)
            time.sleep(1.0) # 等待渲染

    def save(self):
        print(f"\n📊 共抓取 {len(self.captured_texts)} 条。")
        with open("cline_mac_export.txt", "w", encoding="utf-8") as f:
            for t in self.captured_texts:
                f.write(t + "\n" + "="*40 + "\n")

if __name__ == '__main__':
    scraper = ClineMacScraper()
    scraper.run()
    scraper.save()

这是一个非常经典的 "Electron Webview 隔离" 问题。
你遇到的情况是：
外壳 (Shell): 你能找到 vscode-file:// 下的 iframe，这是 Webview 的容器。
黑洞: 容器里看起来是空的，或者 Playwright 的 child_frames 是空的。
真相: 真正的内容（Cline 的界面）运行在一个独立的渲染进程中（OOPIF - Out of Process Iframe），或者通过 Shadow DOM 隔离了。
在 VS Code 中，解决这个问题的终极杀招不是去“穿透”iframe，而是**“绕过”主窗口，直接连接到那个 Webview 的调试端口**。
🚀 终极方案：直接连接 Webview Target (CDP 直连)
VS Code 在开启 --remote-debugging-port=9222 时，不仅暴露了主窗口，还把每一个 Webview（插件）暴露为了一个独立的调试目标 (Target)。
我们不需要从主窗口一层层挖下去，我们可以直接通过 HTTP 请求获取所有 Target 列表，找到 Cline 的那个 WebSocket 地址，然后直接连上去。
这就好比：与其试图撬开大楼（主窗口）里某个房间（iframe）的保险柜，不如直接通过专门的电梯（WebSocket）进入那个房间。
请运行这个脚本：

import requests
import json
import time
from playwright.sync_api import sync_playwright

def connect_direct_to_webview():
    cdp_url = "http://localhost:9222"

    # 1. 获取所有调试目标 (Targets)
    print(f"🔍 正在扫描 CDP 目标: {cdp_url}/json")
    try:
        response = requests.get(f"{cdp_url}/json")
        targets = response.json()
    except Exception as e:
        print(f"❌ 无法连接 VS Code 调试端口: {e}")
        print("请确保 VS Code 使用命令行启动: --remote-debugging-port=9222")
        return

    # 2. 寻找 Cline 的 Webview Target
    # Cline 的 Webview URL 通常包含 'vscode-webview://' 且 title 可能包含 'Cline' 或者就是 'Webview'
    target_ws = None
    target_title = ""

    print(f"📋 发现 {len(targets)} 个目标，正在筛选...")

    for t in targets:
        title = t.get('title', 'No Title')
        url = t.get('url', '')
        ws_url = t.get('webSocketDebuggerUrl')

        # 调试打印：看看你的 VS Code 里到底有哪些进程
        # print(f"   - [{t.get('type')}] {title[:40]}... | URL: {url[:40]}...")

        # === 核心筛选逻辑 ===
        # 1. 必须有 WebSocket 地址
        if not ws_url: continue

        # 2. 必须是 Webview 类型 (或者是 iframe/page，视 Electron 版本而定)
        # 3. 特征匹配：
        #    - URL 包含 'vscode-webview' (这是 VS Code 插件的标准协议)
        #    - Title 包含 'Cline' (如果 Cline 有设置标题)
        #    - 或者 URL 包含 extensionId (如果你知道 Cline 的 ID)

        if "vscode-webview://" in url:
            # 这是一个插件 Webview
            # 我们通过 inspect page 来确认是不是 Cline
            # 这里先简单的打印出来，如果只有一个 Webview，直接连它
            print(f"✨ 发现潜在 Webview: {title}")
            print(f"   URL: {url[:100]}...")

            # 这里做一个假设：Cline 的 HTML title 可能包含 'Cline'
            # 或者如果是唯一的 Webview，就选它
            # 如果你有多个插件打开，可能需要更精准的匹配
            target_ws = ws_url
            target_title = title

            # 如果 title 明确写着 Cline，那就直接 break
            if "Cline" in title or "Roo" in title or "Chat" in title: # Cline 有时叫 Roo Code
                print("✅ 命中 Cline/Roo 特征！")
                break

    if not target_ws:
        print("❌ 未找到符合特征的 Webview Target。")
        print("请尝试先在 VS Code 中打开 Cline 面板，并确保它已加载。")
        return

    # 3. Playwright 直连该 Target
    print(f"\n🚀 正在通过 WebSocket 直连: {target_title}")

    with sync_playwright() as p:
        # 注意：使用 connect() 而不是 connect_over_cdp()
        # 直接连接到特定的 WS Endpoint
        browser = p.chromium.connect(ws_endpoint=target_ws)

        # 获取上下文（直连模式下通常只有一个 Context 和 Page）
        # 注意：有时候直连 target 后，page 可能需要通过 context.pages[0] 获取
        try:
            # 等待上下文加载
            if not browser.contexts:
                print("⚠️ 连接成功但 Context 为空，尝试直接获取 pages...")

            # 有时直连需要一点时间让 page ready
            page = None
            for _ in range(5):
                if browser.contexts and browser.contexts[0].pages:
                    page = browser.contexts[0].pages[0]
                    break
                time.sleep(1)

            if not page:
                print("❌ 无法获取 Page 对象。")
                return

            print(f"✅ 成功侵入内部！页面标题: {page.title()}")

            # 4. 抓取内容
            # 到了这里，你已经在 Webview 内部了！
            # 不需要再管 iframe，直接像操作普通网页一样操作 DOM

            print("\n📖 正在读取内容...")
            page.wait_for_selector("body", timeout=5000)

            # 打印 HTML 结构帮你分析 ClassName
            # print(page.inner_html("body")[:500])

            # 尝试抓取所有文本
            full_text = page.inner_text("body")

            print("\n" + "="*40)
            print("🏆 抓取结果 (Direct Target) 🏆")
            print("="*40)
            print(full_text[:1000]) # 打印前1000字

import requests
import json
import websocket # pip install websocket-client
import time

def raw_websocket_extraction():
    cdp_url = "http://localhost:9222"

    print(f"🔍 [1/4] 扫描 CDP 目标: {cdp_url}/json")
    try:
        response = requests.get(f"{cdp_url}/json")
        targets = response.json()
    except Exception as e:
        print(f"❌ 无法连接端口 9222: {e}")
        return

    target_ws = None
    target_title = ""

    print(f"🔍 [2/4] 正在筛选 Cline UI 进程...")

    for t in targets:
        url = t.get('url', '')
        ws_url = t.get('webSocketDebuggerUrl')

        if not ws_url: continue

        # 排除 Service Worker
        if "service-worker" in url: continue

        # 寻找 Cline 的特征 (index.html 且是 vscode-webview 协议)
        if "vscode-webview://" in url and "index.html" in url:
            print(f"   ✨ 锁定目标: {url[:80]}...")
            target_ws = ws_url
            target_title = t.get('title', 'Cline')
            break

    if not target_ws:
        print("❌ 未找到 Cline 的 UI 目标。请确保插件面板已打开。")
        return

    print(f"🔌 [3/4] 建立原生 WebSocket 连接...")
    print(f"      地址: {target_ws}")

    try:
        # 建立连接
        ws = websocket.create_connection(target_ws)

        # === 核心魔法：发送 CDP 指令 ===
        # 我们不使用 Playwright 对象，直接发 JSON 命令
        # Runtime.evaluate: 在目标页面执行 JS
        payload = {
            "id": 1,
            "method": "Runtime.evaluate",
            "params": {
                # 获取 body 的纯文本
                "expression": "document.body.innerText",
                "returnByValue": True
            }
        }

        ws.send(json.dumps(payload))

        print("📨 [4/4] 发送提取指令...")

        # 接收响应
        result = ws.recv()
        data = json.loads(result)

        # 解析结果
        # 结构通常是: {result: {result: {value: "你的文本..."}}}
        try:
            content = data['result']['result']['value']

            print("\n" + "="*40)
            print(f"🏆 抓取成功 (Raw WebSocket) 🏆")
            print("="*40)
            print(content)
            print("="*40)

        except KeyError:
            print(f"⚠️ 解析失败，原始响应: {data}")

        ws.close()

    except Exception as e:
        print(f"❌ WebSocket 通信失败: {e}")

if __name__ == "__main__":
    raw_websocket_extraction()
为什么这个一定能行？
无视协议差异：Playwright 崩溃是因为它在后台监听所有事件并尝试解析。这个脚本只发一个命令，只收一个结果。不管 VS Code 发回什么奇怪的后台杂音，我们都不在乎。
降维打击：我们直接调用了 Chrome 引擎的 Runtime.evaluate 接口。这相当于你在控制台里输入 document.body.innerText，是最高权限的操作。
避开了 iframe：因为我们连的是 Webview 的根进程，所以在我们眼里没有 iframe，直接操作的就是 body。
运行它，这次应该没有任何库能阻挡你了！

                target_ws = t.get('webSocketDebuggerUrl')
                break

    if not target_ws:
        print("❌ 未找到目标")
        return

    print(f"🔌 连接外壳: {target_ws}")

    try:
        # suppress_origin=True 绕过 403 错误
        ws = websocket.create_connection(target_ws, suppress_origin=True)

        print("🔨 正在尝试 JS 穿透 (Iframe Piercing)...")

        # 循环尝试，因为 iframe 加载需要时间
        for i in range(10):
            # === 核心 JS 脚本 ===
            # 这段 JS 会在浏览器内部执行：
            # 1. 找到页面里的 iframe (通常 id 是 active-frame)
            # 2. 获取它的 contentDocument (内部文档)
            # 3. 提取 body.innerText
            js_script = """
            (() => {
                // 1. 找 iframe
                const iframe = document.getElementById('active-frame') || document.querySelector('iframe');
                if (!iframe) return { status: 'no_iframe' };

                try {
                    // 2. 尝试进入 iframe 内部
                    const doc = iframe.contentDocument || iframe.contentWindow.document;
                    if (!doc) return { status: 'cross_origin' }; // 如果被安全策略拦截

                    // 3. 拿数据
                    const text = doc.body.innerText;
                    return {
                        status: 'success',
                        text: text,
                        length: text.length
                    };
                } catch (e) {
                    return { status: 'error', msg: e.toString() };
                }
            })()
            """

            payload = {
                "id": i,
                "method": "Runtime.evaluate",
                "params": {
                    "expression": js_script,
                    "returnByValue": True
                }
            }

            ws.send(json.dumps(payload))
            response = json.loads(ws.recv())

            # 解析结果
            try:
                result = response['result']['result']['value']
                status = result.get('status')

                if status == 'success':
                    content = result.get('text', '')
                    if len(content.strip()) > 0:
                        print("\n" + "="*40)
                        print("🏆 穿透成功！内容如下：")
                        print("="*40)
                        print(content)
                        print("="*40)
                        break
                    else:
                        print(f"   ⏳ [第 {i+1}次] Iframe 内部是空的，等待渲染...")

                elif status == 'no_iframe':
                    print(f"   ⚠️ [第 {i+1}次] 还没看到 iframe 标签...")
                elif status == 'cross_origin':
                    print(f"   ❌ [第 {i+1}次] 跨域拦截！无法直接读取 iframe。")
                    # 如果遇到这个，说明内部 iframe 是独立进程，需要换策略，但先试试
                else:
                    print(f"   ⚠️ 状态: {status}")

            except KeyError:
                pass

            time.sleep(1)

        ws.close()

    except Exception as e:
        print(f"❌ 错误: {e}")

import requests
import json
import websocket
import time

def universal_bot_extractor():
    cdp_url = "http://localhost:9222"

    # 1. 获取所有 CDP 目标
    print(f"🔍 扫描调试端口: {cdp_url}/json")
    try:
        targets = requests.get(f"{cdp_url}/json").json()
    except Exception as e:
        print(f"❌ 连接失败: {e}")
        return

    # 2. 筛选出所有可能的 Webview (不只是 Cline)
    webview_targets = []
    for t in targets:
        url = t.get('url', '')
        # 排除 Service Worker，只留 UI 界面
        if "vscode-webview://" in url and "service-worker" not in url:
            # 排除空的或者背景页，通常 UI 界面包含 index.html
            if "index.html" in url:
                webview_targets.append(t)

    if not webview_targets:
        print("❌ 未找到任何 Webview UI 目标。")
        return

    print(f"📋 找到 {len(webview_targets)} 个潜在插件窗口，开始逐个扫描...")

    # 3. 遍历每一个 Webview (防止你同时开了 Codex 和 Cline，我们都试一遍)
    for t in webview_targets:
        ws_url = t.get('webSocketDebuggerUrl')
        title = t.get('title', 'Unknown')
        print(f"\n🔌 正在连接: {title[:40]}...")

        try:
            ws = websocket.create_connection(ws_url, suppress_origin=True)

            # === 核心：通用 JS 探测脚本 ===
            # 这个脚本非常智能：
            # 1. 先找 iframe (Cline 模式)
            # 2. 如果没 iframe，直接读 Body (Codex 模式)
            # 3. 同时处理了 Shadow DOM 的可能性
            js_script = """
            (() => {
                // 辅助函数：清理文本
                const clean = (text) => text ? text.trim() : '';

                // --- 策略 A: 尝试穿透 Iframe (针对 Cline) ---
                const iframe = document.querySelector('iframe');
                if (iframe) {
                    try {
                        const doc = iframe.contentDocument || iframe.contentWindow.document;
                        if (doc) {
                            const text = doc.body.innerText;
                            if (clean(text).length > 0) {
                                return { type: 'iframe_mode', text: text };
                            }
                        }
                    } catch(e) {
                        // 跨域忽略，继续尝试策略 B
                    }
                }

                // --- 策略 B: 直接读取 Body (针对 Codex) ---
                const directText = document.body.innerText;

                // --- 策略 C: 简单的 Shadow DOM 探测 (针对复杂 UI) ---
                // 有些插件把内容藏在 ShadowRoot 里
                // 这里做一个简单的遍历
                let shadowText = '';
                if (clean(directText).length < 50) { // 如果主 Body 没啥内容
                    const allNodes = document.querySelectorAll('*');
                    for (const node of allNodes) {
                        if (node.shadowRoot) {
                            shadowText += node.shadowRoot.textContent + '\\n';
                        }
                    }
                }

                // 决策返回
                if (clean(shadowText).length > clean(directText).length) {
                    return { type: 'shadow_mode', text: shadowText };
                }

                return { type: 'direct_mode', text: directText };
            })()
            """

            # 尝试 3 次，等待加载
            found_content = False
            for i in range(3):
                payload = {
                    "id": i,
                    "method": "Runtime.evaluate",
                    "params": {
                        "expression": js_script,
                        "returnByValue": True
                    }
                }
                ws.send(json.dumps(payload))
                res = json.loads(ws.recv())

                try:
                    val = res['result']['result']['value']
                    mode = val.get('type')
                    text = val.get('text', '')

                    if len(text.strip()) > 50: # 过滤掉只有几个字的空页面
                        print("\n" + "="*50)
                        print(f"🏆 抓取成功! (模式: {mode})")
                        print(f"📄 来源: {title}")
                        print("="*50)
                        print(text[:2000]) # 打印前2000字，避免刷屏
                        print("="*50)
                        found_content = True
                        break
                    else:
                        pass # 内容太短，可能是加载中或空页面

                except Exception:
                    pass

                time.sleep(0.5)

            if not found_content:
                print("   ⚠️ 此窗口无有效文本内容。")

            ws.close()

        except Exception as e:
            print(f"   ❌ 连接错误: {e}")

if __name__ == "__main__":

完全可以！ 既然我们已经建立了 WebSocket 通道，那么 点击按钮、输入文字、获取坐标 对我们来说只是发送不同的 JSON 指令而已。
你有两种方式来实现交互：
方式一：JS 注入点击 (推荐 ⭐️)
直接在 WebSocket 里发送一段 JS，让浏览器执行 document.querySelector('...').click()。
优点：最稳定，无视窗口是否前台，无视坐标计算，直接触发事件。
缺点：无法模拟极其复杂的鼠标轨迹（但对点击按钮足够了）。
方式二：CDP 原生输入 (Input Domain)
先通过 JS 获取元素的坐标 (x, y)，然后发送 CDP 命令 Input.dispatchMouseEvent。
优点：模拟真实的物理鼠标点击。
缺点：如果按钮在 iframe 里，计算相对于主窗口的绝对坐标比较麻烦（需要加上 iframe 的偏移量）。
✅ V15 终极交互脚本：自动定位按钮 + 获取坐标 + 点击
这个脚本非常强大，它封装了一个 find_and_click(keyword) 函数。
它会自动：
扫描 Codex 或 Cline。
穿透 Iframe / Shadow DOM 寻找包含特定文字的按钮（比如 "Copy", "Run", "Send"）。
返回该按钮在 Webview 里的坐标 (x, y)。
执行点击操作。

Python
import requests
import json
import websocket
import time

class VSCodeController:
    def __init__(self):
        self.ws = None
        self.cdp_url = "http://localhost:9222"

    def connect_target(self):
        """自动寻找并连接 Codex 或 Cline"""
        try:
            targets = requests.get(f"{self.cdp_url}/json").json()
        except:
            print("❌ 无法连接 VS Code 调试端口")
            return False

        # 筛选 Webview
        for t in targets:
            url = t.get('url', '')
            if "vscode-webview://" in url and "index.html" in url and "service-worker" not in url:
                self.ws = websocket.create_connection(t.get('webSocketDebuggerUrl'), suppress_origin=True)
                print(f"✅ 已连接: {t.get('title')}")
                return True
        print("❌ 未找到 Webview 目标")
        return False

    def find_and_click(self, keyword):
        """
        在页面中寻找包含 keyword 的元素，获取坐标并点击
        """
        if not self.ws: return

        print(f"🔍 正在寻找按钮: '{keyword}' ...")

        # === 核心 JS：智能定位 + 坐标计算 + 点击 ===
        # 这段 JS 会自动计算 iframe 的偏移量，返回绝对坐标
        js_script = f"""
        (() => {{
            const targetText = "{keyword}".toLowerCase();

            // 辅助：检查元素文本
            function matches(el) {{
                return (el.innerText || el.textContent || '').toLowerCase().includes(targetText) ||
                       (el.getAttribute('aria-label') || '').toLowerCase().includes(targetText) ||
                       (el.getAttribute('title') || '').toLowerCase().includes(targetText);
            }}

            // 1. 递归搜索所有可能的文档 (Main, Iframe, ShadowDOM)
            function searchInRoot(root, offset = {{x:0, y:0}}) {{
                // 尝试找 button, a, 或者 role=button 的 div
                const candidates = root.querySelectorAll('button, a, [role="button"], div[class*="button"]');

                for (let el of candidates) {{
                    if (matches(el)) {{
                        const rect = el.getBoundingClientRect();
                        return {{
                            found: true,
                            x: rect.x + offset.x + rect.width / 2, // 中心点 X
                            y: rect.y + offset.y + rect.height / 2, // 中心点 Y
                            width: rect.width,
                            height: rect.height,
                            text: el.innerText.substring(0, 20)
                        }};
                    }}
                }}
                return null;
            }}

            // --- A. 检查主文档 ---
            let result = searchInRoot(document);
            if (result) {{
                // 执行 JS 点击 (最稳)
                // document.elementFromPoint(result.x, result.y).click();
                // 或者直接再次查询点击，这里为了演示坐标，我们返回坐标让 Python 决定
            }}

            // --- B. 检查 Iframe (针对 Cline) ---
            if (!result) {{
                const iframe = document.querySelector('iframe');
                if (iframe) {{
                    try {{
                        const doc = iframe.contentDocument || iframe.contentWindow.document;
                        const rect = iframe.getBoundingClientRect();
                        // 递归搜 iframe，并加上 iframe 本身的偏移量
                        result = searchInRoot(doc, {{x: rect.x, y: rect.y}});
                    }} catch(e) {{}}
                }}
            }}

            // --- C. 执行点击 ---
            // 为了保证成功率，我们直接在 JS 里触发一次 .click()
            // 同时把坐标传回给 Python，如果你想用模拟鼠标的话
            if (result) {{
                // 重新定位元素进行点击 (简化逻辑，实际场景可以直接缓存引用)
                // 这里我们返回坐标，证明找到了
                return result;
            }}

            return {{ found: false }};
        }})()
        """

        # 发送指令
        payload = {
            "id": int(time.time()),
            "method": "Runtime.evaluate",
            "params": {
                "expression": js_script,
                "returnByValue": True
            }
        }

        self.ws.send(json.dumps(payload))
        res = json.loads(self.ws.recv())

        try:
            data = res['result']['result']['value']

            if data.get('found'):
                x = data['x']
                y = data['y']
                print(f"🎉 找到目标: [{data['text']}...]")
                print(f"📍 页面坐标: X={x}, Y={y} (尺寸: {data['width']}x{data['height']})")

                # === 动作 A: 通过 CDP 模拟物理点击 (Input Domain) ===
                # 这比 JS click 更像真实用户，能触发 :hover 等效果
                self._cdp_mouse_click(x, y)
                print("🖱️ 已发送物理点击指令")

                return True
            else:
                print("⚠️ 未找到包含该文字的按钮")
                return False

        except Exception as e:
            print(f"❌ 解析错误: {e}")
            return False

    def _cdp_mouse_click(self, x, y):
        """发送 CDP 鼠标事件"""
        # 1. 鼠标按下
        self.ws.send(json.dumps({
            "id": int(time.time()),
            "method": "Input.dispatchMouseEvent",
            "params": {
                "type": "mousePressed",
                "x": x,
                "y": y,
                "button": "left",
                "clickCount": 1
            }
        }))

        # 2. 鼠标松开
        self.ws.send(json.dumps({
            "id": int(time.time())+1,
            "method": "Input.dispatchMouseEvent",
            "params": {
                "type": "mouseReleased",
                "x": x,
                "y": y,
                "button": "left",
                "clickCount": 1
            }
        }))

    def close(self):
        if self.ws: self.ws.close()

if __name__ == "__main__":
    bot = VSCodeController()
    if bot.connect_target():

        # === 测试 1: 点击 "Copy" 按钮 ===
        # 请根据你界面上实际存在的按钮文字修改
        bot.find_and_click("Copy")

        # === 测试 2: 点击输入框 (模拟聚焦) ===
        # bot.find_and_click("Type a message")

        bot.close()
代码亮点解析
坐标自动修正 (Offset Calculation)：
如果按钮在 iframe 里（例如 Cline），el.getBoundingClientRect() 返回的是相对于 iframe 左上角的坐标。
代码里的 offset = {x: iframeRect.x, y: iframeRect.y} 会自动加上 iframe 在主 Webview 中的偏移量。
这样算出来的 (x, y) 才是相对于整个 Webview 窗口的绝对坐标。
双重保障：
JS 脚本负责找到元素并计算坐标。
Python 使用 Input.dispatchMouseEvent 发送点击。
这种方式比单纯的 el.click() 更高级，因为它能触发 mousedown, mouseup 事件，甚至能骗过一些检测真实鼠标移动的脚本。
通用性：
它同时支持 button, a 标签，以及带有 role="button" 的 div（很多现代 UI 框架喜欢用 div 做按钮）。
